# Проблемы текущей архитектуры и риски при масштабировании

1. **Синхронные вызовы** между сервисами создают жёсткие зависимости, таймауты и точки отказа.
2. **Локальное кэширование данных о продуктах** приводит к рассогласованию состояния между `core-app` и `ins-comp-settlement`.
3. **Ежедневная выгрузка данных** через REST создаёт пиковую нагрузку и уязвима к недоступности сервисов.
4. При подключении дополнительных страховых компаний **нагрузка на `ins-product-aggregator` удвоится**, что усугубит текущие проблемы.
5. **Отсутствует реактивное обновление**: клиенты видят устаревшие данные о тарифах.

## Предлагаемое решение
- Ввести **единый `product-catalog-service`** как источник истины.
- Заменить синхронные вызовы на **асинхронные события через Kafka**.
- Использовать **паттерн Transactional Outbox** для надёжной доставки событий.
- Подписаться на события вместо регулярных опросов.
